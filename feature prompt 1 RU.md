## 🧠 ЧТО ТЫ ДОЛЖНА СДЕЛАТЬ

Ты — старший продакт-аналитик и Scrum-мастер. На основе блока «ОПИСАНИЕ ФИЧИ» проведи полное исследование и спланируй работу, строго по шагам:

1. **Исследование фичи.** Разбери цель, ценность для бизнеса и пользователя, ключевые сценарии, граничные случаи, зависимости и риски (включая нормативные/правовые, если применимо).
2. **Роли и персоны.** Выдели все роли, которые касаются фичи: конечные пользователи, операторы, комплаенс/админы и системная роль **«Платформа»** (бэкенд/системные требования). Для каждой — короткое описание и цвет (hex).
3. **Активности (этапы пути).** Сгруппируй работу в этапы пользовательского пути / функциональные блоки. Каждой активности — короткий код (`A1`, `A2`, `B1`, `O1`, `C1`…) и человекочитаемое название.
4. **User stories по INVEST.** Для каждой истории сформулируй три части раздельно: **роль** (как…), **действие** (я хочу…), **ценность** (чтобы…). Добавь 2–5 чётких **критериев приёмки** (acceptance criteria).
5. **Оценка (Story Points).** Оцени каждую историю по шкале **только из значений: 1, 2, 3, 5, 8, 13, 20, 40, 100**. Истории крупнее 13 SP по возможности дроби на меньшие.
6. **Приоритет.** Проставь `P0` (критично/блокер MVP), `P1` (важно), `P2` (желательно).
7. **Зависимости.** В `deps` укажи ID других историй (`US-XX`) или внешние ссылки (нормы/стандарты, внешние сервисы, открытые вопросы — например `Email-сервис`, `Q1`).
8. **Планирование спринтов и сроков.** Разложи истории по спринтам так, чтобы:
   - сумма SP в одном спринте **не превышала capacity** этого спринта;
   - зависимости шли **раньше** зависящих от них историй;
   - `P0` попадали в самые ранние спринты;
   - `release` каждой истории совпадал с релизом её спринта.
   Сгенерируй спринты (`S1`, `S2`, …) с реальными датами, отсчитывая от даты старта с шагом в длину спринта.
9. **Вехи и юр.трек** (если применимо). Добавь ключевые milestones (даты + подпись) и правовые/подготовительные задачи в `legal`.

## 📐 СХЕМА ВЫХОДНОГО JSON (соблюдать строго)

> Примеры ниже — **нейтральные образцы формата** (не привязаны к домену). Значения подставь свои, исходя из фичи. Даты — лишь образец; фактические считаются от «Дата старта» из КОНФИГа.

Верхний уровень — один объект ровно с такими ключами:

```jsonc
{
  "stories":          [ /* массив историй, см. ниже — ОБЯЗАТЕЛЬНО */ ],
  "activities":       [ { "code": "A1", "name": "Онбординг" } ],
  "activityOrder":    [ "A1", "A2", "A3", "A4", "A5", "A6" ],   // порядок отображения, все коды из activities
  "sprints":          [ { "code": "S1", "start": "2026-06-01", "end": "2026-06-14", "release": "MVP", "capacity": 30 } ],
  "personas":         [
                        { "name": "Пользователь", "desc": "Целевой конечный пользователь продукта", "color": "#0EA5E9" },
                        { "name": "Платформа",    "desc": "Бэкенд / системные требования",          "color": "#94A3B8" }
                      ],
  "spHours":          8,                                         // часов в 1 SP
  "milestones":       [ { "date": "2026-09-30", "label": "🚀 Релиз MVP" } ],
  "legal":            [ { "code": "L1", "name": "Подготовка юр. документации", "start": "2026-06-29", "end": "2026-07-26", "critical": true } ],
  "versions":         [],
  "currentVersionId": null,
  "exportedAt":       "2026-06-01T00:00:00.000Z"                 // ISO-таймстамп генерации
}
```

**Объект истории (элемент `stories[]`) — нейтральный пример формата:**

```jsonc
{
  "id":           "US-04",                                 // уникальный, префикс из конфига
  "activity":     "A1",                                    // код, обязан существовать в activities[]
  "activityName": "Онбординг",                             // ДОЛЖЕН совпадать с name этой активности
  "persona":      "Пользователь",                          // обязан совпадать с одним из personas[].name
  "role":         "новым пользователем",                   // часть «как …»
  "action":       "зарегистрироваться по email",           // часть «я хочу …»
  "value":        "быстро начать пользоваться продуктом",  // часть «чтобы …»
  "ac": [                                                  // критерии приёмки, 2–5 шт.
    "Форма принимает email и пароль",
    "Подтверждение по ссылке из письма",
    "Профиль создаётся автоматически",
    "Уведомление об успешной регистрации"
  ],
  "sp":           8,                                       // одно из: 1,2,3,5,8,13,20,40,100
  "sprint":       "S1",                                    // код, обязан существовать в sprints[]
  "release":      "MVP",                                   // ровно одно из: MVP | R1 | R2
  "priority":     "P0",                                    // ровно одно из: P0 | P1 | P2
  "deps":         [],                                      // массив строк: "US-XX" или внешние ссылки
  "status":       "todo",                                  // НЕОБЯЗАТЕЛЬНО: todo | in_progress | done (по умолч. todo)
  "tfsPbi":       []                                       // НЕОБЯЗАТЕЛЬНО: массив строк (номера PBI)
}
```

## ✅ ЖЁСТКИЕ ПРАВИЛА ВАЛИДАЦИИ (проверь перед выводом)

1. `priority` — только `P0`/`P1`/`P2`. `release` — только `MVP`/`R1`/`R2`. `status` — только `todo`/`in_progress`/`done`.
2. `sp` — только из набора `1,2,3,5,8,13,20,40,100`. Никаких других чисел.
3. Каждый `story.activity` присутствует в `activities[]`, а `activityName` дословно равен его `name`.
4. Каждый `story.persona` присутствует в `personas[]` (`name`).
5. Каждый `story.sprint` присутствует в `sprints[]`, и `story.release` равен `release` этого спринта.
6. Все коды из `activities[]` перечислены в `activityOrder[]` в логичном порядке.
7. Сумма `sp` всех историй в каждом спринте **≤** `capacity` этого спринта.
8. Все `id` уникальны; ссылки в `deps` на `US-XX` указывают на реально существующие истории.
9. Даты — формат `YYYY-MM-DD`. `color` персон — hex (`#RRGGBB`).
10. `versions: []`, `currentVersionId: null`, `exportedAt` — валидный ISO-таймстамп.
